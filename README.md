# TheGamingRoom
Portfolio project containing the software design document and analysis for the Draw It or Lose It application developed for The Gaming Room.

Working on the software design document for The Gaming Room gave me a clearer understanding of how a client's needs can turn into a working design. The client wanted to redevelop and expand their existing game, Draw It or Lose It, from an Android only application into a web-based system that could support multiple platforms and simultaneous users. The overall goal was to design a system that improves accessibility, supports multiple users, and ensures consistent performance with the structure of the original game. This required thinking about business requirements, such as supporting multiple teams and players, and technical requirements like only allowing one instance of the game to exist at a time. 

I focused on designing the structure of the game so that each section connected to the client's needs. The way that the requirements, constraints, and recommendations are arranged makes the design clearer and easy to follow. The evaluation section also plays an important role by comparing operating systems in a way that supported my decision to recommend Linux. 

Writing the design document before working on any code was more beneficial than I expected because it required me to think through the structure of the application and how different components would interact. By not jumping straight into coding, I had a clearer understanding of how the application should function as a whole. This made the development itself more efficient because I had already considered the decisions about architecture and system interactions. 

Expanding the system architecture section of this design document would strengthen the overall design. A detailed diagram was not necessary for the purposes of this document, but adding more explanation about the communication and interaction of the components would better complete the design. A visual representation or more detailed description of how the client, server, and database work together would give a better understanding of how the system functions.

Understanding the user's needs was a key role throughout the design process. When turning these needs into system requirements, it was important to make each decision one that supported functionality and user experience. The main component of the design was the need to support multiple players and teams simultaneously while improving access by using a web-based approach to allow the game to work on across different platforms. 

The design process involved breaking the system into smaller components and understanding how each of these individual functions work together made it easier to understand. I used about-oriented principles, such as inheritance and encapsulation, and the Singleton pattern to ensure only one instance of the game exists at a time.  This maintained consistency in the data across the application. This approach included validating names by searching existing data to prevent duplicates. I will continue to use these approaches in the future to keep the design organized and make the overall process clearer. 

Working through this project showed the value of thinking through a design before writing code. A clear plan made the development process more manageable and helped connect the technical decisions back to the client's needs.
